HTTP/HTTPS — how browsers and servers talk

HTTP stands for HyperText Transfer Protocol. It's the set of rules that browsers and servers use to communicate. Every time you open a website, an HTTP request happens — you just never see it.

🔒 Think of it like this: HTTP is the language your browser and the server speak to each other. HTTPS is the same language but with a digital lock — everything is scrambled so nobody can spy on it.

HTTPS is HTTP with encryption. The "S" stands for Secure. Almost all websites today use HTTPS because it protects your passwords, credit card numbers, and other private data.

What an HTTP request looks like

When you type a URL and press Enter, your browser sends something like this:

GET /page.html HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Accept: text/html,application/xhtml+xml Connection: close (optional: request body for POST)

What an HTTP response looks like

The server replies with something like this:

HTTP/1.1 200 OK Content-Type: text/html; charset=utf-8 Content-Length: 1234 Set-Cookie: session=abc123 <html>...the page content...</html>

HTTP methods — what do you want to do?

Method What it does Safe? Has body?
GETGet data from the serverYesNo
POSTSend new data to the serverNoYes
PUTReplace existing dataNoYes
PATCHUpdate part of existing dataNoYes
DELETEDelete data from the serverNoNo

HTTP status codes — what happened?

Every response has a number that tells you if things went well or not:

2xx — Success:
  • 200 OK — Everything worked fine
  • 201 Created — New item was created
  • 204 No Content — Worked fine, but nothing to send back
3xx — Redirected:
  • 301 Moved Permanently — Page has a new permanent address
  • 302 Found — Page is temporarily somewhere else
  • 304 Not Modified — Your cached version is still fine
4xx — You made a mistake:
  • 400 Bad Request — Your request doesn't make sense
  • 401 Unauthorized — You need to log in first
  • 403 Forbidden — You're logged in but not allowed
  • 404 Not Found — That page doesn't exist
5xx — The server made a mistake:
  • 500 Internal Server Error — Something crashed on the server
  • 503 Service Unavailable — Server is down or overloaded

HTTP headers — extra info on every request

Headers are like sticky notes attached to every request and response. They carry extra info:

  • Content-Type: What kind of data this is (HTML, JSON, image, etc.)
  • Content-Length: How big the data is
  • Cache-Control: Can the browser save this for later?
  • Authorization: A token proving you're logged in
  • Set-Cookie: Store a small piece of data in the browser
  • Access-Control-Allow-Origin: Which websites are allowed to use this data (CORS)

HTTP/2 and HTTP/3

Newer versions of HTTP are faster. HTTP/2 can send many requests at once. HTTP/3 is even faster, using a newer protocol called QUIC.

DNS — the internet's phone book

DNS stands for Domain Name System. Without DNS, you couldn't type "google.com" into your browser — you'd have to know the actual IP address (like 142.250.74.46). DNS translates domain names into IP addresses that computers can use.

📕 Think of it like this: DNS is like a phone book for the internet. You look up a name (google.com) and it tells you the number (the IP address). Then your browser can make the call.

DNS is one of the most important systems on the internet. Billions of DNS lookups happen every single day. If DNS goes down, huge parts of the internet stop working.

How DNS works, step by step

1. Browser checks its own cache first
↓
2. Asks the Recursive Resolver (usually your ISP)
↓
3. Resolver asks a Root Nameserver
↓
4. Root says: "Ask the TLD nameserver for .com"
↓
5. TLD says: "Ask the authoritative nameserver for that domain"
↓
6. Authoritative server gives back the IP address
↓
7. Browser gets the IP and connects!

DNS record types

  • A Record: Links a domain to an IPv4 address (like 192.0.2.1)
  • AAAA Record: Links a domain to an IPv6 address (the newer, longer format)
  • CNAME: An alias — "this domain is the same as that one"
  • MX Record: Which server handles email for this domain
  • TXT Record: Text info — used for verification, spam protection, etc.
  • NS Record: Which nameserver is responsible for this domain

DNS caching — saving lookups for later

DNS lookups take time, so results are saved (cached) at several points:

  • Browser cache: Your browser remembers recent lookups for a few minutes
  • OS cache: Your computer also saves lookups
  • ISP cache: Your internet provider caches them too
  • TTL (Time To Live): How long a result can be cached — set by the domain owner

DNS security

  • DNSSEC: Adds digital signatures to prevent fake DNS responses
  • DoH (DNS over HTTPS): Encrypts DNS lookups so no one can spy on which sites you visit
  • DoT (DNS over TLS): Same idea, different method

APIs — how apps talk to each other

API stands for Application Programming Interface. An API is a set of rules for how two programs can communicate. One program says "give me this", and the API defines exactly what comes back.

🍽️ Think of it like this: An API is like a menu in a restaurant. You pick what you want from the menu (the API). The kitchen (the server) makes it and sends it out. You don't need to know how it was cooked — just what to order.

APIs are everywhere. When your weather app shows the forecast, it's calling a weather API. When you log in with Google, that's OAuth — a type of API. When a map loads in an app, that's the Google Maps API.

Why are APIs useful?

  • Abstraction: You don't need to understand the internals — just the interface
  • Reusability: Many apps can use the same API
  • Security: You only expose what you want to share
  • Flexibility: The inside can change without breaking the outside
  • Standard: Everyone speaks the same language

REST APIs — the most common style

REST (Representational State Transfer) is the most popular way to design web APIs. A good REST API follows these ideas:

  • Resources: Everything is a "thing" you can address, like /users or /posts
  • HTTP Methods: GET to read, POST to create, PUT/PATCH to update, DELETE to remove
  • Stateless: Each request contains all the info needed — no memory between requests
  • JSON: Data is usually sent and received as JSON
  • Status codes: Use the right HTTP status code for each response

A real example — Twitter API

GET /2/tweets/:id → Get a specific tweet POST /2/tweets → Post a new tweet DELETE /2/tweets/:id → Delete a tweet GET /2/users/:id/following → Get who this user follows

Authentication — proving who you are

Most APIs require you to prove your identity before they'll respond:

  • API Keys: A secret code you include in every request
  • OAuth 2.0: The standard used by Google, Facebook, GitHub, etc.
  • JWT (JSON Web Tokens): A small token that proves who you are
  • HTTP Basic Auth: Username and password (only safe over HTTPS)

GraphQL — a newer alternative

GraphQL lets you ask for exactly the fields you need — no more, no less. This avoids downloading data you don't use:

query { user(id: 1) { name email posts { title date } } }

Rate limiting — don't overload the server

Most APIs limit how many requests you can make. If you go over the limit, you get a 429 error. A typical limit might be 1000 requests per hour.
