Servers & hosting

Servers — the computers that run the web

A server is a computer whose job is to wait for requests and respond to them. When your browser makes an HTTP request, it's talking to a server. The server runs the code, reads from the database, and sends back a response. Everything in this section — PHP, JSON, databases — lives and runs on a server.

🍕 Think of it like this: You're ordering pizza. The pizza place (the server) is always open, waiting for orders. You call up and place your order (the HTTP request), and they send the pizza back (the HTTP response). The pizza place doesn't call you — it waits for you to call first. And the kitchen — with all the ingredients, equipment, and recipes — is the backend you never see.

Servers are normal computers — they just run all day, every day, and have a fast internet connection. They live in data centres: large buildings with row upon row of computers, kept cool and connected to the internet at high speed.

VPS — renting your own server

Unless you work at a big tech company, you don't usually own a server. You rent one. The most common way is a VPS — a Virtual Private Server.

A VPS takes one large physical server and splits it into many smaller virtual servers. Each one acts like its own independent computer, with its own operating system, storage, and network connection. You install PHP, a database, a web server like nginx — whatever your app needs — and it runs around the clock.

  • Shared hosting: Cheapest option. Many websites share the same server. Limited control. Fine for simple sites.
  • VPS: Your own slice of a server. More control, better performance. Good for most web apps.
  • Dedicated server: An entire physical machine just for you. Expensive, but maximum power.
  • Cloud (AWS, Google Cloud, Azure): Rent as many virtual servers as you need and scale up or down instantly.
💡 Good places to start: DigitalOcean, Hetzner, and Vultr are popular VPS providers that developers recommend for their simplicity and fair pricing. A basic VPS costs around $4–6 per month.

PHP — programming on the server

PHP stands for Hypertext Preprocessor. The big difference from JavaScript is that PHP runs on the server, not in your browser. That means:

  • Visitors can never see your PHP code — they only ever get HTML back
  • You can safely connect to databases
  • You can do heavy work on the server before sending a response
  • You can show different content to different users
👨‍🍳 Think of it like this: PHP is like a chef in a restaurant. You (the browser) place your order (the request). The chef works in the kitchen (the server — fully hidden). You get your finished meal (the response). You never see what happened in the kitchen.

How PHP works, step by step

Browser sends a request
↓
Server receives it
↓
Server runs the PHP code
↓
PHP builds an HTML page
↓
Server sends that HTML to the browser
↓
Browser shows the page

PHP + HTML = dynamic pages

PHP can generate different HTML depending on who's visiting or what time it is:

<?php $currentTime = date('H:i:s'); $user = "Alex"; if ($currentTime < "12:00:00") { echo "Good morning, " . $user; } else { echo "Good afternoon, " . $user; } ?>

Security — this is really important!

Because PHP runs on the server and often talks to databases, bad security can cause serious damage. Here are the most common dangers:

⚠️ SQL Injection: A hacker types database commands into a form field to steal or destroy your data.
// WRONG — dangerous! $sql = "SELECT * FROM users WHERE email = '" . $email . "'"; // CORRECT — safe with prepared statements $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?"); $stmt->execute([$email]);
  • XSS (Cross-Site Scripting): Malicious code injected into your page — use htmlspecialchars()
  • CSRF: Fake form submissions that trick logged-in users — use tokens
  • Passwords: Never store them as plain text — always hash them with password_hash()
  • File uploads: Always check the file type and size

PHP best practices today

  • Use a framework: Laravel or Symfony — don't write everything from scratch
  • Use prepared statements: To prevent SQL injection
  • Validate all input: Never trust what users send
  • Escape output: So HTML characters don't cause problems
  • Always use HTTPS: Never send data unencrypted

JSON — the web's standard data format

JSON stands for JavaScript Object Notation. It's a simple way to write structured data as text. The cool thing about JSON is that it works with every programming language — it's not tied to JavaScript at all.

📦 Think of it like this: JSON is like a standard shipping box. No matter who packed it or who receives it, everyone knows how to open it and read what's inside. Any programming language can read JSON.

JSON is everywhere on the web. When Instagram loads your feed, when Google Maps shows you directions, when a weather app shows the forecast — that data almost certainly arrives as JSON.

Why use JSON?

  • Easy to read: Both humans and computers can understand it
  • Compact: Takes up less space than older formats like XML
  • Flexible: You can nest data inside data
  • Universal: Supported by every modern language and platform
  • Fast: Quick to parse and generate

JSON data types

JSON can hold these types of values:

{ "string": "text", "number": 42, "decimal": 3.14, "boolean": true, "null_value": null, "array": [1, 2, 3], "object": { "nested": "value" } }

A real example — what an API might send back

{ "status": "success", "data": { "id": 123, "name": "Alex", "email": "alex@example.com", "loggedIn": true, "interests": ["web", "games"], "license": { "premium": true, "expires_on": "2025-12-31" } } }

Databases — storing data permanently

A database is an organized way to store lots of data so you can find it again quickly. Your user account on any website, your saved posts, your order history — all of that lives in a database.

📚 Think of it like this: A database is like a super-fast, super-smart spreadsheet. You can store millions of rows and search through them in milliseconds. Try that in Excel!

SQL vs NoSQL — two different approaches

There are two main types of databases, and they work very differently:

SQL (Relational databases) — structured and strict:
  • Structure: Tables with rows and columns, like a spreadsheet
  • Schema: You must define the columns before you can add data
  • ACID: Transactions are reliable and safe
  • Examples: MySQL, PostgreSQL, SQLite, SQL Server
  • Good for: Bank accounts, orders, anything where accuracy matters
NoSQL (Document databases) — flexible and fast:
  • Structure: Documents (think JSON objects)
  • Schema: Flexible — each document can have different fields
  • Scalable: Easy to spread across many servers
  • Examples: MongoDB, Firebase, CouchDB
  • Good for: User profiles, content, logs, anything that changes a lot

SQL in action

-- Get all users SELECT * FROM users; -- Filter and sort SELECT name, email FROM users WHERE city = 'Amsterdam' ORDER BY name; -- Count users per city SELECT city, COUNT(*) FROM users GROUP BY city; -- Join two tables together SELECT users.name, posts.title FROM users JOIN posts ON users.id = posts.user_id;

Indexes — making searches faster

An index is like the index at the back of a book. Instead of reading every page to find something, you jump straight to it. Indexes make database searches much faster, but they take up extra space.

  • Primary Key: A unique ID for every row
  • Foreign Key: A link to a row in another table
  • Trade-off: Faster to read, a little slower to write

Database security

  • Backups: Always back up your data!
  • Access control: Not everyone should be able to read or change everything
  • Encryption: Encrypt sensitive data like passwords
  • Parameterized queries: Protect against SQL injection attacks
