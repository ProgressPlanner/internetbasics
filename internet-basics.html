HTML — de structuur van het web

HTML staat voor HyperText Markup Language. Elke website die je ooit hebt bezocht is gebouwd met HTML. Het is de basis van het web. HTML vertelt de browser wat elk onderdeel op een pagina is — een koptekst, een paragraaf, een afbeelding, een knop.

🏠 Vergelijking: Als een website een huis zou zijn, is HTML de muren, ramen, deuren en kamers. Het bepaalt de indeling en structuur — niet hoe het eruitziet of wat het doet.

Als je een website opent, downloadt je browser het HTML-bestand van de server. Dat bestand bevat alle tekst en vertelt de browser waar hij afbeeldingen en andere bestanden kan vinden. De browser leest dit en bouwt de pagina die jij ziet.

Jouw eerste HTML-pagina

Elk HTML-bestand begint met hetzelfde basisraamwerk. Zie het als de omhulling die je altijd om je content heen bouwt:

<!DOCTYPE html> <html lang="nl"> <head> <meta charset="UTF-8"> <title>Mijn Pagina</title> </head> <body> <h1>Hallo, wereld!</h1> <p>Dit is mijn eerste webpagina.</p> </body> </html>
  • <!DOCTYPE html> — Vertelt de browser dat dit een moderne HTML5-pagina is
  • <html lang="nl"> — De buitenste omhulling. lang vertelt schermlezers welke taal het is
  • <head> — Instelling-informatie die de browser nodig heeft, maar die je niet ziet
  • <meta charset="UTF-8"> — Zorgt ervoor dat speciale tekens zoals é en ü correct worden weergegeven
  • <title> — De tekst die in het browsertabblad verschijnt
  • <body> — Alles wat zichtbaar is op de pagina komt hier

Waarom is HTML handig?

  • Universeel: Elk apparaat met een browser kan HTML lezen
  • Betekenisvol: HTML-tags beschrijven wat iets is, niet alleen hoe het eruitziet
  • Makkelijk om mee te starten: Je leert de basis in een middag
  • Goed voor zoekmachines: Google kan HTML lezen en begrijpen
  • Toegankelijk: Schermlezers (voor blinde gebruikers) lezen HTML hardop voor

Semantische HTML — de juiste tag voor de juiste inhoud

Moderne HTML heeft speciale tags die beschrijven wat de inhoud betekent, niet alleen hoe het eruitziet. Dit helpt Google en schermlezers je pagina te begrijpen:

  • <header> — Het bovenste deel van een pagina of sectie
  • <nav> — Een navigatiemenu
  • <main> — De hoofdinhoud van de pagina
  • <article> — Een blogartikel of zelfstandig stuk inhoud
  • <section> — Een groep gerelateerde inhoud
  • <aside> — Een zijbalk of extra informatie
  • <footer> — De onderkant van een pagina

Coole dingen die HTML5 kan

Moderne HTML5 (de nieuwste versie) heeft veel ingebouwde mogelijkheden:

  • Video en audio: Speel video's en muziek af met <video> en <audio>
  • Canvas: Teken vormen en animaties direct in de browser
  • SVG: Schaalbare afbeeldingen die scherp blijven op elk formaat
  • Formuliervalidatie: Controleer of een e-mailadres er geldig uitziet vóór het formulier wordt verstuurd
  • Geolocatie: Vraag de gebruiker om zijn locatie
  • LocalStorage: Sla kleine stukjes data op in de browser van de gebruiker

Tags die je altijd gebruikt

Er zijn honderden HTML-tags, maar een kleine groep dekt bijna alles af wat je ooit nodig hebt:

<!-- Koppelingen --> <a href="https://example.com">Klik hier</a> <!-- Afbeeldingen --> <img src="foto.jpg" alt="Een hond in het park"> <!-- Ongeordende lijst --> <ul> <li>Eerste item</li> <li>Tweede item</li> </ul> <!-- Formulierinvoer --> <input type="text" placeholder="Typ hier..."> <button>Verstuur</button>

CSS — maak het mooi

CSS staat voor Cascading Style Sheets. HTML vertelt de browser wat iets is. CSS vertelt hem hoe het eruit moet zien. Kleuren, lettertypen, ruimte, lay-out, animaties — dat is allemaal CSS.

🎨 Vergelijking: Als HTML de muren en kamers van een huis zijn, is CSS de verf, meubels en decoratie. Het bepaalt hoe alles eruitziet en aanvoelt.

Het woord "Cascading" betekent dat stijlen naar beneden vloeien en elkaar kunnen overschrijven. Als je een regel instelt en daarna een nieuwe, wint de nieuwe (meestal).

Hoe een CSS-regel eruitziet

Elke CSS-regel heeft drie onderdelen: een selector (wat je wilt opmaken), een eigenschap (wat je wilt veranderen) en een waarde (wat je het op instelt):

selector { eigenschap: waarde; } ────────── ───────────────────────── │ │ └── Hoeveel? Welke kleur? Welk lettertype? │ └─────────────── Wat wil je veranderen? └──────────────────────────── Welke HTML-elementen wil je aanspreken?
p { color: #333333; font-size: 16px; line-height: 1.6; }

Eenheden — hoe groot is groot?

CSS heeft meerdere manieren om groottes uit te drukken. De juiste kiezen maakt een verschil:

  • px — Pixels. Vaste grootte. Goed voor randen en kleine precisie-elementen.
  • em — Relatief ten opzichte van de lettergrootte van het bovenliggende element.
  • rem — Relatief ten opzichte van de root-lettergrootte (meestal 16px). Voorspelbaarder dan em.
  • % — Een percentage van de grootte van het bovenliggende element. Ideaal voor flexibele lay-outs.
  • vw / vh — Een percentage van de schermbreedte of -hoogte. 50vw = de helft van de schermbreedte.
💡 Denk eraan zo: px is als zeggen "precies 30 centimeter" — vast, ongeacht de context. rem is als zeggen "twee keer de normale tekstgrootte" — het past zich automatisch aan als de gebruiker de tekst groter maakt in de browserinstellingen, wat goed is voor toegankelijkheid.

CSS selectoren — kiezen wat je wilt opmaken

Een selector kiest welke HTML-elementen een stijl krijgen:

/* Alle paragrafen opmaken */ p { color: blue; } /* Elementen met class="highlight" */ .highlight { background: yellow; } /* Het element met id="header" */ #header { font-size: 24px; } /* Selectoren combineren */ p.intro { font-weight: bold; } /* Een link bij hoveren */ a:hover { color: red; } button:active { transform: scale(0.95); }

Specificiteit — welke regel wint?

Als twee CSS-regels hetzelfde element aanspreken, wint de meest specifieke. Zie het als een puntensysteem:

  • ID-selector #header — 100 punten. Sterkst.
  • Klasse-selector .highlight — 10 punten.
  • Elementselector p — 1 punt. Zwakst.
/* 1 punt — past groen toe */ p { color: green; } /* 10 punten — overschrijft met blauw */ .intro { color: blue; } /* 100 punten — overschrijft alles met rood */ #main-text { color: red; }
💡 Tip: Gebruik het liefst klasse-selectors. Te veel ID's gebruiken maakt CSS later moeilijk te overschrijven en te onderhouden.

Het boxmodel — alles is een doos

Elk element op een pagina wordt behandeld als een doos met vier lagen:

MARGIN — ruimte buiten het element
├─ BORDER — de rand
│ ├─ PADDING — ruimte tussen rand en inhoud
│ │ └─ CONTENT — de tekst of afbeelding zelf

Flexbox en Grid — moderne lay-outtools

Deze twee CSS-functies maken het veel makkelijker om dingen op een pagina te plaatsen:

  • Flexbox: Zet dingen op een rij of in een kolom
  • Grid: Maakt een volledig tabelachtige lay-out met rijen én kolommen

Media queries — andere stijlen voor andere schermen

Met media queries schrijf je CSS die alleen op bepaalde schermformaten geldt. Zo maak je een site die er goed uitziet op zowel een telefoon als een laptop:

@media (max-width: 768px) { body { font-size: 14px; } .sidebar { display: none; } } @media (prefers-color-scheme: dark) { body { background: #333; color: white; } }

JavaScript — maak het interactief

JavaScript is een echte programmeertaal die direct in de browser draait. Het maakt webpagina's interactief. Zonder JavaScript zouden websites statisch zijn — alleen tekst en afbeeldingen zonder knoppen, animaties of live updates.

🧠 Vergelijking: Als HTML de structuur is en CSS het uiterlijk, dan is JavaScript het brein. Het zorgt dat dingen gebeuren en reageren op wat jij doet — een knop indrukken, een formulier invullen, een video afspelen.

JavaScript begon als een kleine taal voor browsers, maar draait nu overal. Servers (Node.js), mobiele apps (React Native), desktopapps (Electron) — allemaal JavaScript. Het is waarschijnlijk de populairste programmeertaal ter wereld.

Variabelen — informatie opslaan

Een variabele is een benaamd bakje dat een waarde bewaart. Gebruik const voor waarden die niet veranderen, en let voor waarden die dat wel doen:

const naam = 'Alex'; // tekst — een string const leeftijd = 16; // een getal const ingelogd = true; // waar of niet waar — een boolean let score = 0; // kan later veranderen score = score + 10; // nu is score 10 const kleuren = ['rood', 'groen', 'blauw']; // een lijst — een array const gebruiker = { naam: 'Alex', leeftijd: 16 }; // sleutel-waardeparen — een object

Functies — herbruikbare blokken code

Een functie is een benaamd blok code dat je kunt aanroepen wanneer je maar wilt:

function begroet(naam) { return 'Hallo, ' + naam + '!'; } console.log(begroet('Alex')); // print: Hallo, Alex! console.log(begroet('Sam')); // print: Hallo, Sam!

Beslissingen nemen — if en else

Gebruik if om code alleen uit te voeren als een voorwaarde waar is:

const score = 75; if (score >= 55) { console.log('Geslaagd!'); } else { console.log('Probeer het opnieuw.'); }

Wat kan JavaScript doen?

  • Reageren op klikken, typen, scrollen — alles wat jij doet
  • De pagina aanpassen zonder hem opnieuw te laden
  • Data opslaan in de browser (zoals je winkelmandje)
  • Data ophalen van servers (zoals nieuwe berichten laden)
  • Controleren of een formulier goed is ingevuld vóór het versturen
  • Animaties en visuele effecten maken
  • Games bouwen (denk aan games die in de browser draaien!)
  • Realtime chatten (zoals WhatsApp Web)

De DOM — hoe JavaScript met HTML praat

Als een browser je HTML laadt, bouwt hij er een boom van objecten van. Die boom heet de DOM (Document Object Model). JavaScript kan elk onderdeel van die boom vinden, lezen en aanpassen — zo maak je pagina's interactief.

🌳 Denk eraan zo: De DOM is een stamboom voor je HTML. <html> is de wortel. <head> en <body> zijn de kinderen. Alles in <body> vertakt zich daar vandaan. JavaScript kan elk knooppunt bereiken, lezen, aanpassen of nieuwe toevoegen.
// Vind een element aan de hand van het id-attribuut const titel = document.getElementById('hoofd-titel'); titel.textContent = 'Aangepast door JavaScript!'; // querySelector werkt net als een CSS-selector const knop = document.querySelector('.mijn-knop'); // Luister naar events — zoals een klik op een knop knop.addEventListener('click', function() { titel.textContent = 'Je hebt op de knop geklikt!'; });

Populaire JavaScript-frameworks

Voor grote projecten gebruiken developers frameworks — kant-en-klare tools die veel sneller werken:

  • React: Gemaakt door Facebook — heel veel gebruikt voor interfaces
  • Vue.js: Iets eenvoudiger dan React, goed voor beginners
  • Angular: Gemaakt door Google — groot en krachtig
  • Svelte: Nieuwer, heel snel, minder code nodig
  • Next.js: React met extra functies zoals server-side rendering

TypeScript — JavaScript met vangrails

TypeScript is JavaScript met één grote extra functie: types. Je geeft aan wat voor soort data elke variabele mag bevatten, en TypeScript waarschuwt je direct als er iets niet klopt — voordat de code ook maar wordt uitgevoerd.

🛡️ Denk eraan zo: JavaScript laat je van alles in elk bakje stoppen. TypeScript plakt labels op de bakjes — "alleen getallen", "alleen tekst". Als je er per ongeluk het verkeerde in stopt, pikt TypeScript dat meteen op.
// JavaScript — geen types, geen waarschuwingen let score = 100; score = "oeps, nu een string"; // JavaScript vindt dit prima // TypeScript — types toegevoegd let score: number = 100; score = "oeps, nu een string"; // ❌ TypeScript waarschuwt meteen // Functies met getypeerde parameters en returnwaarde function begroet(naam: string): string { return 'Hallo, ' + naam; }

TypeScript draait niet rechtstreeks in de browser — het wordt eerst omgezet (gecompileerd) naar gewoon JavaScript. De meeste grote frameworks (React, Vue, Angular) werken naadloos samen met TypeScript, en de meeste professionele teams gebruiken het.

💡 Moet je TypeScript leren? Begin eerst met JavaScript. Als je vertrouwd bent met variabelen, functies en de DOM, is TypeScript veel makkelijker op te pikken. Het is zeker de moeite waard voordat je je eerste baan als webontwikkelaar begint.

Async JavaScript — dingen doen zonder te wachten

Als je browser data opvraagt bij een server, bevriest hij niet. Hij blijft gewoon draaien en verwerkt het antwoord zodra het binnenkomt. Dit heet asynchrone code:

// Promises — overzichtelijker dan diep geneste callbacks fetch('/api/data') .then(response => response.json()) .then(data => console.log(data)); // Async/await (modern en het schoonst) async function haalData() { const response = await fetch('/api/data'); const data = await response.json(); return data; }

Servers & hosting

Servers — de computers die het web draaiende houden

Een server is een computer waarvan de taak het is om op verzoeken te wachten en ze te beantwoorden. Als jouw browser een HTTP-verzoek doet, praat hij met een server. De server voert de code uit, leest uit de database en stuurt een antwoord terug. Alles in dit hoofdstuk — PHP, JSON, databases — draait op een server.

🍕 Denk eraan zo: Je bestelt pizza. De pizzeria (de server) is altijd open en wacht op bestellingen. Jij belt op en plaatst je bestelling (het HTTP-verzoek), en zij sturen de pizza terug (het HTTP-antwoord). De pizzeria belt jou niet — ze wachten totdat jij belt. En de keuken — met alle ingrediënten, apparatuur en recepten — is de backend die je nooit ziet.

Servers zijn gewone computers — ze draaien gewoon dag en nacht en hebben een snelle internetverbinding. Ze staan in datacenters: grote gebouwen vol rijen computers, goed gekoeld en aangesloten op het internet met hoge snelheid.

VPS — jouw eigen server huren

Tenzij je bij een groot techbedrijf werkt, bezit je doorgaans geen server. Je huurt er één. De meest gebruikelijke manier is een VPS — een Virtual Private Server.

Een VPS neemt één grote fysieke server en verdeelt die in meerdere kleinere virtuele servers. Elke VPS gedraagt zich als een eigen onafhankelijke computer, met een eigen besturingssysteem, opslag en netwerkverbinding. Je installeert PHP, een database, een webserver zoals nginx — wat je app ook nodig heeft — en het draait dag en nacht.

  • Gedeelde hosting: Goedkoopste optie. Meerdere websites delen dezelfde server. Weinig controle. Prima voor eenvoudige sites.
  • VPS: Jouw eigen stukje van een server. Meer controle, betere prestaties. Goed voor de meeste webapps.
  • Dedicated server: Een complete fysieke machine alleen voor jou. Duur, maar maximale kracht.
  • Cloud (AWS, Google Cloud, Azure): Huur zoveel virtuele servers als je nodig hebt en schaal direct op of af.
💡 Goede startpunten: DigitalOcean, Hetzner en Vultr zijn populaire VPS-aanbieders die geliefd zijn bij ontwikkelaars vanwege hun eenvoud en eerlijke prijzen. Een eenvoudige VPS kost ongeveer €4–6 per maand.

PHP — programmeren op de server

PHP staat voor Hypertext Preprocessor. Het grote verschil met JavaScript: PHP draait op de server, niet in je browser. Dat betekent:

  • Bezoekers kunnen je PHP-code nooit zien — ze krijgen alleen HTML terug
  • Je kunt veilig verbinding maken met databases
  • Je kunt zwaar werk op de server doen voordat je antwoord geeft
  • Je kunt verschillende inhoud tonen aan verschillende gebruikers
👨‍🍳 Vergelijking: PHP is als een kok in een restaurant. Jij (de browser) geeft je bestelling door (het verzoek). De kok werkt in de keuken (de server — volledig verborgen). Jij krijgt je afgewerkt gerecht (het antwoord). Je ziet nooit wat er in de keuken is gebeurd.

Hoe PHP werkt, stap voor stap

Browser stuurt een verzoek
↓
Server ontvangt het
↓
Server voert de PHP-code uit
↓
PHP bouwt een HTML-pagina
↓
Server stuurt die HTML naar de browser
↓
Browser toont de pagina

PHP + HTML = dynamische pagina's

PHP kan verschillende HTML genereren afhankelijk van wie er op bezoek is of hoe laat het is:

<?php $huidigeTijd = date('H:i:s'); $gebruiker = "Alex"; if ($huidigeTijd < "12:00:00") { echo "Goedemorgen, " . $gebruiker; } else { echo "Goedemiddag, " . $gebruiker; } ?>

Beveiliging — dit is heel belangrijk!

Omdat PHP op de server draait en vaak met databases communiceert, kan slechte beveiliging grote schade aanrichten. Dit zijn de meest voorkomende gevaren:

⚠️ SQL Injectie: Een hacker typt database-commando's in een formulierveld om jouw data te stelen of te verwijderen.
// FOUT — gevaarlijk! $sql = "SELECT * FROM users WHERE email = '" . $email . "'"; // CORRECT — veilig met prepared statements $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?"); $stmt->execute([$email]);
  • XSS (Cross-Site Scripting): Kwaadaardige code in je pagina injecteren — gebruik htmlspecialchars()
  • CSRF: Nep-formulieren die ingelogde gebruikers misleiden — gebruik tokens
  • Wachtwoorden: Nooit als gewone tekst opslaan — altijd hashen met password_hash()
  • Bestandsuploads: Controleer altijd het bestandstype en de grootte

Moderne PHP-aanpak

  • Gebruik een framework: Laravel of Symfony — schrijf niet alles zelf
  • Gebruik prepared statements: Beschermt tegen SQL-injectie
  • Valideer alle invoer: Vertrouw nooit wat gebruikers sturen
  • Codeer uitvoer: Zodat HTML-tekens geen problemen veroorzaken
  • Gebruik altijd HTTPS: Stuur nooit data onversleuteld

JSON — het standaard dataformaat van het web

JSON staat voor JavaScript Object Notation. Het is een eenvoudige manier om gestructureerde data als tekst op te schrijven. Het leuke aan JSON is dat het werkt met elke programmeertaal — het is niet gebonden aan JavaScript.

📦 Vergelijking: JSON is als een standaard verzenddoos. Het maakt niet uit wie hem heeft ingepakt of wie hem ontvangt — iedereen weet hoe hij hem moet openen en lezen. Elke programmeertaal kan JSON lezen.

JSON is overal op het web. Als Instagram je tijdlijn laadt, als Google Maps je route toont, als een weer-app de voorspelling weergeeft — die data komt vrijwel altijd aan als JSON.

Waarom JSON gebruiken?

  • Makkelijk te lezen: Zowel mensen als computers begrijpen het
  • Compact: Neemt minder ruimte in dan oudere formaten zoals XML
  • Flexibel: Je kunt data in data stoppen
  • Universeel: Elke moderne taal en elk platform ondersteunt het
  • Snel: Snel te verwerken en te maken

JSON-gegevenstypen

JSON kan deze soorten waarden bevatten:

{ "tekst": "hallo", "getal": 42, "decimaal": 3.14, "boolean": true, "null_waarde": null, "array": [1, 2, 3], "object": { "genest": "waarde" } }

Een echt voorbeeld — wat een API terugstuur

{ "status": "success", "data": { "id": 123, "naam": "Alex", "email": "alex@voorbeeld.nl", "ingelogd": true, "interesses": ["web", "games"], "licentie": { "premium": true, "verlopen_op": "2025-12-31" } } }

HTTP/HTTPS — hoe browsers en servers praten

HTTP staat voor HyperText Transfer Protocol. Het is de set regels die browsers en servers gebruiken om met elkaar te communiceren. Elke keer dat je een website opent, gebeurt er een HTTP-verzoek — je ziet het alleen nooit.

🔒 Vergelijking: HTTP is de taal die je browser en de server met elkaar spreken. HTTPS is dezelfde taal maar met een digitaal hangslot — alles is versleuteld zodat niemand kan meekijken.

HTTPS is HTTP met versleuteling. De "S" staat voor Secure (veilig). Vrijwel alle websites gebruiken tegenwoordig HTTPS, omdat het je wachtwoorden, creditcardnummers en andere privédata beschermt.

Hoe een HTTP-verzoek eruitziet

Als je een URL intypt en op Enter drukt, stuurt je browser zoiets als dit:

GET /pagina.html HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Accept: text/html,application/xhtml+xml Connection: close (optioneel: verzoekinhoud bij POST)

Hoe een HTTP-antwoord eruitziet

De server antwoordt met zoiets als dit:

HTTP/1.1 200 OK Content-Type: text/html; charset=utf-8 Content-Length: 1234 Set-Cookie: session=abc123 <html>...de pagina-inhoud...</html>

HTTP-methoden — wat wil je doen?

Methode Wat het doet Veilig? Met inhoud?
GETData ophalen van de serverJaNee
POSTNieuwe data naar de server sturenNeeJa
PUTBestaande data vervangenNeeJa
PATCHEen deel van bestaande data bijwerkenNeeJa
DELETEData van de server verwijderenNeeNee

HTTP-statuscodes — wat is er gebeurd?

Elk antwoord heeft een getal dat vertelt of het goed is gegaan:

2xx — Gelukt:
  • 200 OK — Alles werkte prima
  • 201 Created — Nieuw item aangemaakt
  • 204 No Content — Gelukt, maar niets terug te sturen
3xx — Doorgestuurd:
  • 301 Moved Permanently — Pagina heeft een nieuw permanent adres
  • 302 Found — Pagina staat tijdelijk ergens anders
  • 304 Not Modified — Je gecachte versie is nog steeds geldig
4xx — Jij hebt een fout gemaakt:
  • 400 Bad Request — Je verzoek klopt niet
  • 401 Unauthorized — Je moet eerst inloggen
  • 403 Forbidden — Je bent ingelogd maar hebt geen toegang
  • 404 Not Found — Die pagina bestaat niet
5xx — De server heeft een fout gemaakt:
  • 500 Internal Server Error — Er is iets gecrasht op de server
  • 503 Service Unavailable — Server is down of overbelast

HTTP-headers — extra info bij elk verzoek

Headers zijn als plakbriefjes die bij elk verzoek en antwoord zitten. Ze geven extra informatie mee:

  • Content-Type: Wat voor data dit is (HTML, JSON, afbeelding, etc.)
  • Content-Length: Hoe groot de data is
  • Cache-Control: Mag de browser dit bewaren voor later?
  • Authorization: Een token dat bewijst dat je bent ingelogd
  • Set-Cookie: Sla een klein stukje data op in de browser
  • Access-Control-Allow-Origin: Welke websites mogen deze data gebruiken (CORS)

HTTP/2 en HTTP/3

Nieuwere versies van HTTP zijn sneller. HTTP/2 kan meerdere verzoeken tegelijk versturen. HTTP/3 is nog sneller dankzij een nieuwer protocol genaamd QUIC.

DNS — het telefoonboek van het internet

DNS staat voor Domain Name System. Zonder DNS zou je niet "google.com" in je browser kunnen typen — je zou het echte IP-adres moeten weten (zoals 142.250.74.46). DNS vertaalt domeinnamen naar IP-adressen die computers kunnen gebruiken.

📕 Vergelijking: DNS is als een telefoonboek voor het internet. Je zoekt een naam op (google.com) en het geeft je het nummer (het IP-adres). Dan kan je browser bellen (verbinden).

DNS is een van de belangrijkste systemen van het internet. Elke dag vinden er miljarden DNS-opzoekingen plaats. Als DNS uitvalt, liggen grote delen van het internet plat.

Hoe DNS werkt, stap voor stap

1. Browser controleert zijn eigen cache eerst
↓
2. Vraagt de Recursive Resolver (meestal je internetprovider)
↓
3. Resolver vraagt een Root Nameserver
↓
4. Root zegt: "Vraag de TLD-nameserver voor .com"
↓
5. TLD zegt: "Vraag de authoritative nameserver voor dat domein"
↓
6. Authoritative server geeft het IP-adres terug
↓
7. Browser krijgt het IP-adres en maakt verbinding!

DNS-recordtypen

  • A Record: Koppelt een domein aan een IPv4-adres (zoals 192.0.2.1)
  • AAAA Record: Koppelt een domein aan een IPv6-adres (het nieuwere, langere formaat)
  • CNAME: Een alias — "dit domein is hetzelfde als dat domein"
  • MX Record: Welke server e-mail afhandelt voor dit domein
  • TXT Record: Tekstinfo — gebruikt voor verificatie, spambescherming, etc.
  • NS Record: Welke nameserver verantwoordelijk is voor dit domein

DNS-caching — opzoekingen bewaren voor later

DNS-opzoekingen kosten tijd, dus worden resultaten opgeslagen (gecacht) op meerdere plekken:

  • Browsercache: Je browser onthoudt recente opzoekingen een paar minuten
  • OS-cache: Je computer slaat ze ook op
  • ISP-cache: Je internetprovider slaat ze ook op
  • TTL (Time To Live): Hoe lang een resultaat gecacht mag worden — ingesteld door de domeineigenaar

DNS-beveiliging

  • DNSSEC: Voegt digitale handtekeningen toe om neppe DNS-antwoorden te voorkomen
  • DoH (DNS over HTTPS): Versleutelt DNS-opzoekingen zodat niemand kan zien welke sites je bezoekt
  • DoT (DNS over TLS): Zelfde idee, andere methode

APIs — hoe apps met elkaar praten

API staat voor Application Programming Interface. Een API is een set regels voor hoe twee programma's kunnen communiceren. Het ene programma zegt "geef mij dit", en de API bepaalt precies wat er terugkomt.

🍽️ Vergelijking: Een API is als een menu in een restaurant. Je kiest wat je wilt uit het menu (de API). De keuken (de server) maakt het en stuurt het eruit. Je hoeft niet te weten hoe het is bereid — alleen wat je kunt bestellen.

APIs zijn overal. Als je weer-app de voorspelling toont, gebruikt die een weer-API. Als je inlogt met Google, is dat OAuth — een soort API. Als een kaart laadt in een app, is dat de Google Maps API.

Waarom zijn APIs handig?

  • Abstractie: Je hoeft het inwendige niet te begrijpen — alleen de interface
  • Herbruikbaar: Veel apps kunnen dezelfde API gebruiken
  • Veilig: Je deelt alleen wat je wilt delen
  • Flexibel: De binnenkant kan veranderen zonder de buitenkant te breken
  • Standaard: Iedereen spreekt dezelfde taal

REST APIs — de meest gebruikte stijl

REST (Representational State Transfer) is de populairste manier om web-APIs te ontwerpen. Een goede REST API volgt deze ideeën:

  • Resources: Alles is een "ding" dat je kunt aanspreken, zoals /gebruikers of /berichten
  • HTTP-methoden: GET om te lezen, POST om aan te maken, PUT/PATCH om bij te werken, DELETE om te verwijderen
  • Stateless: Elk verzoek bevat alle benodigde info — geen geheugen tussen verzoeken
  • JSON: Data wordt gewoonlijk verstuurd en ontvangen als JSON
  • Statuscodes: Gebruik de juiste HTTP-statuscode bij elk antwoord

Een echt voorbeeld — Twitter API

GET /2/tweets/:id → Haal een specifieke tweet op POST /2/tweets → Plaats een nieuwe tweet DELETE /2/tweets/:id → Verwijder een tweet GET /2/users/:id/following → Wie volgt deze gebruiker

Authenticatie — bewijzen wie je bent

De meeste APIs vereisen dat je je identiteit bewijst voordat ze antwoorden:

  • API Keys: Een geheime code die je bij elk verzoek meestuurt
  • OAuth 2.0: De standaard die Google, Facebook en GitHub gebruiken
  • JWT (JSON Web Tokens): Een klein token dat bewijst wie je bent
  • HTTP Basic Auth: Gebruikersnaam en wachtwoord (alleen veilig via HTTPS)

GraphQL — een modern alternatief

GraphQL laat je vragen om precies de velden die je nodig hebt — niet meer, niet minder. Zo download je geen data die je toch niet gebruikt:

query { user(id: 1) { naam email posts { titel datum } } }

Rate limiting — de server niet overbelasten

De meeste APIs beperken hoeveel verzoeken je kunt doen. Als je over de limiet gaat, krijg je een 429-fout. Een typische limiet is 1000 verzoeken per uur.

Databases — data permanent opslaan

Een database is een georganiseerde manier om veel data op te slaan zodat je het later snel terug kunt vinden. Je gebruikersaccount op elke website, je opgeslagen berichten, je bestelhistorie — dat alles staat in een database.

📚 Vergelijking: Een database is als een supersnelle, superschrandere spreadsheet. Je kunt miljoenen rijen opslaan en er in milliseconden doorheen zoeken. Probeer dat maar eens in Excel!

SQL vs NoSQL — twee verschillende aanpakken

Er zijn twee hoofdtypen databases, en ze werken heel anders:

SQL (Relationele databases) — gestructureerd en strikt:
  • Structuur: Tabellen met rijen en kolommen, zoals een spreadsheet
  • Schema: Je moet de kolommen definiëren voordat je data kunt toevoegen
  • ACID: Transacties zijn betrouwbaar en veilig
  • Voorbeelden: MySQL, PostgreSQL, SQLite, SQL Server
  • Goed voor: Bankrekeningen, bestellingen, alles waarbij nauwkeurigheid telt
NoSQL (Document-databases) — flexibel en snel:
  • Structuur: Documenten (denk aan JSON-objecten)
  • Schema: Flexibel — elk document kan andere velden hebben
  • Schaalbaar: Makkelijk te verdelen over meerdere servers
  • Voorbeelden: MongoDB, Firebase, CouchDB
  • Goed voor: Gebruikersprofielen, content, logs, alles wat veel verandert

SQL in de praktijk

-- Alle gebruikers ophalen SELECT * FROM gebruikers; -- Filteren en sorteren SELECT naam, email FROM gebruikers WHERE stad = 'Amsterdam' ORDER BY naam; -- Gebruikers per stad tellen SELECT stad, COUNT(*) FROM gebruikers GROUP BY stad; -- Twee tabellen samenvoegen SELECT users.naam, posts.titel FROM users JOIN posts ON users.id = posts.user_id;

Indexes — zoekopdrachten versnellen

Een index is als de index achter in een boek. In plaats van elke bladzijde te lezen om iets te vinden, spring je er direct naartoe. Indexes maken databasezoekopdrachten veel sneller, maar nemen wel extra ruimte in.

  • Primary Key: Een uniek ID voor elke rij
  • Foreign Key: Een link naar een rij in een andere tabel
  • Afweging: Sneller lezen, iets langzamer schrijven

Databasebeveiliging

  • Back-ups: Maak altijd een back-up van je data!
  • Toegangscontrole: Niet iedereen mag alles lezen of wijzigen
  • Versleuteling: Versleutel gevoelige data zoals wachtwoorden
  • Parameterized queries: Beschermt tegen SQL-injectie-aanvallen

Frontend vs backend — de twee helften van een web-app

Elke professionele webapplicatie heeft twee hoofdonderdelen. Ze draaien op verschillende computers en praten met elkaar via het internet.

🏪 Vergelijking: Frontend is de winkelvloer — wat klanten zien en mee omgaan. Backend is het magazijn — waar al het echte werk plaatsvindt, volledig buiten zicht.

Frontend — wat in je browser draait

Technologieën: HTML, CSS, JavaScript, React, Vue, Angular
Draait op: Jouw apparaat (telefoon, laptop, etc.)
Wat het doet:
  • Toont de interface — knoppen, menu's, tekst, afbeeldingen
  • Reageert direct op klikken en typen, zonder te wachten
  • Controleert formulierinvoer voordat die naar de server gaat
  • Stuurt verzoeken naar de backend
  • Toont de ontvangen data (gewoonlijk JSON)
  • Kan offline werken met service workers
Voordelen:
  • Snel — reageert direct zonder de server te vragen
  • Kan offline werken
  • Minder belasting op de server
Beperkingen:
  • Iedereen kan je JavaScript-code lezen
  • Kan niet rechtstreeks met een database praten
  • Beperkt door het apparaat van de gebruiker

Backend — wat op de server draait

Technologieën: PHP, Python, Node.js, Java, Go, Rust
Draait op: Een server in een datacenter
Wat het doet:
  • Ontvangt verzoeken van de frontend
  • Voert de bedrijfslogica uit (controles, berekeningen, regels)
  • Leest en schrijft naar de database
  • Stuurt JSON terug naar de frontend
  • Houdt code en data privé en veilig
Voordelen:
  • Code is volledig verborgen voor gebruikers
  • Kan schalen — voeg meer servers toe indien nodig
  • Directe toegang tot de database
  • Kan zware taken uitvoeren (e-mails versturen, afbeeldingen verwerken)
Beperkingen:
  • Elke actie vereist een netwerkverzoek (langzamer)
  • Kost geld om te hosten
  • Als het uitvalt, ligt alles plat

De volledige reis — een inlogpoging, stap voor stap

Frontend (Browser)
1. Gebruiker vult het inlogformulier in
2. JavaScript controleert of e-mail en wachtwoord er geldig uitzien
3. JavaScript stuurt HTTPS POST naar de server

↓ (reist over het internet) ↓

Backend (Server)
4. PHP ontvangt het verzoek
5. PHP zoekt het e-mailadres op in de database
6. PHP controleert de wachtwoordhash (nooit gewone tekst opslaan!)
7. PHP maakt een sessietoken
8. PHP stuurt JSON terug: { "success": true, "token": "..." }

↓ (reist over het internet) ↓

Frontend (Browser)
9. JavaScript leest de JSON
10. JavaScript slaat het token op
11. JavaScript gaat naar het dashboard
12. Gebruiker ziet zijn dashboard!

De meeste moderne apps zijn "API-First" gebouwd: de backend bouwt een nette, gedocumenteerde API. De website, mobiele app en externe apps gebruiken allemaal dezelfde API. Zo hoeft de logica niet dubbel te worden geschreven.

Responsive design — één website, alle schermen

Responsive design betekent dat je website er goed uitziet en werkt op elk scherm — een kleine telefoon (320px breed), een tablet, een laptop, een groot beeldscherm. Meer dan 60% van het webverkeer komt tegenwoordig van telefoons, dus dit is echt belangrijk.

🪑 Vergelijking: Responsive design is als verstelbare autostoelen. Dezelfde stoel past een klein kind én een lange volwassene. Een responsieve website past zich net zo makkelijk aan elk schermformaat aan.

De drie pijlers van responsive design

1. Flexibele lay-outs
  • Gebruik percentages in plaats van vaste pixelbreedtes
  • Gebruik CSS Grid of Flexbox voor de opmaak
  • Elementen groeien en krimpen mee met het scherm
2. Flexibele media
  • Afbeeldingen passen automatisch (max-width: 100%)
  • Video's passen zich ook aan aan het schermformaat
  • Gebruik srcset om hoge resolutie-afbeeldingen alleen naar hoge resolutie-schermen te sturen
3. Media Queries
  • Schrijf andere CSS voor verschillende schermformaten
  • Veelgebruikte breekpunten: 480px (telefoon), 768px (tablet), 1024px (desktop)
  • Kan ook landscape versus portretoriëntatie aanpakken

Mobile-first — begin klein, bouw op

De moderne aanpak is om eerst voor mobiel te ontwerpen en dan stijlen toe te voegen voor grotere schermen:

/* MOBIEL — de basisstijlen */ body { font-size: 14px; padding: 10px; } .container { display: block; /* verticaal stapelen op kleine schermen */ } /* TABLET */ @media (min-width: 768px) { body { font-size: 16px; } .container { display: grid; grid-template-columns: 1fr 1fr; } } /* DESKTOP */ @media (min-width: 1024px) { body { font-size: 18px; max-width: 1200px; margin: 0 auto; } .sidebar { width: 250px; } }

Responsive best practices

  • Viewport meta-tag: Voeg altijd <meta name="viewport" content="width=device-width"> toe
  • Geschikt voor aanraken: Knoppen moeten minimaal 44x44px zijn — vingers zijn minder precies dan muiscursors
  • Leesbare tekst: Gebruik minimaal 16px op mobiel
  • Afbeeldingen comprimeren: Mobiele verbindingen zijn trager — stuur geen grote bestanden
  • Testen op echte apparaten: Emulatoren zijn goed, maar echte telefoons zijn beter
  • Geen horizontaal scrollen: Als je pagina zijdelings scrolt, klopt er iets niet

Tools voor responsive testen

  • Browser DevTools: Druk op F12, klik dan op het telefoonpictogram om verschillende schermen te simuleren
  • Google Mobile-Friendly Test: Controleer of Google je pagina als mobielgeschikt beschouwt
  • Lighthouse: Ingebouwd in Chrome — test prestaties, toegankelijkheid en SEO
  • BrowserStack: Test op echte apparaten in de cloud
