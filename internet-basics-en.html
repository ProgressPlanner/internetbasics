HTML — the structure of the web

HTML stands for HyperText Markup Language. Every single website you've ever visited is built with HTML. It's the basic building block of the web. HTML tells the browser what each thing on a page is — a heading, a paragraph, an image, a button.

🏠 Think of it like this: If a website were a house, HTML would be the walls, windows, doors and rooms. It decides the layout and structure — not how it looks or what it does.

When you open a website, your browser downloads the HTML file from the server. That file contains all the text and tells the browser where to find images and other files. The browser reads it and builds the page you see.

Your first HTML page

Every HTML file starts with the same basic skeleton. Think of it as the frame you always build around your content:

<!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <title>My Page</title> </head> <body> <h1>Hello, world!</h1> <p>This is my first webpage.</p> </body> </html>
  • <!DOCTYPE html> — Tells the browser this is a modern HTML5 page
  • <html lang="en"> — The outer wrapper. lang tells screen readers what language it is
  • <head> — Setup info the browser needs but the user doesn't see
  • <meta charset="UTF-8"> — Makes sure special characters like é and ü show correctly
  • <title> — The text that appears in the browser tab
  • <body> — Everything visible on the page goes here

Why is HTML useful?

  • Universal: Every device with a browser can read HTML
  • Meaningful: HTML tags describe what something is, not just how it looks
  • Easy to start: You can learn the basics in an afternoon
  • Good for search engines: Google can read and understand HTML
  • Accessible: Screen readers (for blind users) use HTML to read pages out loud

How a browser turns HTML into a page

1. Browser downloads the HTML file
↓
2. Browser reads (parses) the HTML
↓
3. Browser builds a map of the page called the DOM
↓
4. Browser loads CSS to add styling
↓
5. Browser loads JavaScript to add interactivity
↓
6. You see the finished page!

Semantic HTML — using the right tag for the job

Modern HTML has special tags that describe what content means, not just what it looks like. This helps Google and screen readers understand your page:

  • <header> — The top part of a page or section
  • <nav> — A navigation menu
  • <main> — The main content of the page
  • <article> — A blog post or standalone piece of content
  • <section> — A group of related content
  • <aside> — A sidebar or extra info
  • <footer> — The bottom of a page

Cool things HTML5 can do

Modern HTML5 (the latest version) has lots of built-in features:

  • Video and audio: Play videos and music with <video> and <audio>
  • Canvas: Draw shapes and animations directly in the browser
  • SVG: Scalable images that look sharp at any size
  • Form validation: Check if an email address looks right before the form is sent
  • Geolocation: Ask the user for their location
  • LocalStorage: Save small bits of data in the user's browser

Try it yourself!

Tags you'll use all the time

There are hundreds of HTML tags, but a small handful covers almost everything you'll ever need:

<!-- Links --> <a href="https://example.com">Click here</a> <!-- Images --> <img src="photo.jpg" alt="A dog sitting in the park"> <!-- Unordered list --> <ul> <li>First item</li> <li>Second item</li> </ul> <!-- Form input --> <input type="text" placeholder="Type something..."> <button>Submit</button>
💡 The alt attribute on images matters: Screen readers read it aloud to blind users. Always write a real description — not just "image" or "photo". It also shows if the image fails to load, and helps search engines understand what the image is.

CSS — make it look great

CSS stands for Cascading Style Sheets. HTML tells the browser what something is. CSS tells it how it should look. Colors, fonts, spacing, layout, animations — all of that is CSS.

🎨 Think of it like this: If HTML is the walls and rooms of a house, CSS is the paint, furniture and decoration. It decides how everything looks and feels.

The word "Cascading" means that styles flow down and can overwrite each other. If you set a rule and then set another rule later, the later one wins (usually).

How a CSS rule is written

Every CSS rule has three parts: a selector (what to style), a property (what to change), and a value (what to set it to):

selector { property: value; } ────────── ───────────────────── │ │ └── How much? Which colour? Which font? │ └────────────── What do you want to change? └─────────────────────────── Which HTML elements are you targeting?
p { color: #333333; font-size: 16px; line-height: 1.6; }

Units — how big is big?

CSS has several ways to express sizes. The right choice matters:

  • px — Pixels. Fixed size. Good for borders and small precise things.
  • em — Relative to the font size of the parent element. Scales with text.
  • rem — Relative to the root font size (usually 16px). More predictable than em.
  • % — A percentage of the parent's size. Great for flexible layouts.
  • vw / vh — A percentage of the screen width or height. 50vw = half the screen width.
💡 Think of it like this: px is like saying "exactly 30 centimetres" — fixed no matter what. rem is like saying "twice the normal text size" — it automatically adjusts if the user makes text bigger in their browser settings, which is good for accessibility.

CSS selectors — choosing what to style

A selector picks which HTML elements to apply a style to:

/* Style all paragraphs */ p { color: blue; } /* Style elements with class="highlight" */ .highlight { background: yellow; } /* Style the element with id="header" */ #header { font-size: 24px; } /* Combine selectors */ p.intro { font-weight: bold; } /* Style a link when hovered */ a:hover { color: red; } button:active { transform: scale(0.95); }

Specificity — which rule wins?

If two CSS rules target the same element, the more specific one wins. Think of it as a points system:

  • ID selector #header — 100 points. Strongest.
  • Class selector .highlight — 10 points.
  • Element selector p — 1 point. Weakest.
/* 1 point — applies green */ p { color: green; } /* 10 points — overrides with blue */ .intro { color: blue; } /* 100 points — overrides everything with red */ #main-text { color: red; }
💡 Tip: Try to rely on class selectors most of the time. Overusing IDs makes CSS hard to override later and harder for your future self to maintain.

The box model — everything is a box

Every element on a page is treated as a box. The box has four layers:

MARGIN — space outside the element
├─ BORDER — the outline
│ ├─ PADDING — space between border and content
│ │ └─ CONTENT — the actual text or image

Flexbox and Grid — modern layout tools

These two CSS features make it much easier to place things on a page:

  • Flexbox: Lines things up in a row or column
  • Grid: Creates a full table-like layout with rows AND columns

Media queries — different styles for different screens

Media queries let you write CSS that only applies on certain screen sizes. This is how you make a site look good on both a phone and a laptop:

@media (max-width: 768px) { body { font-size: 14px; } .sidebar { display: none; } } @media (prefers-color-scheme: dark) { body { background: #333; color: white; } }

JavaScript — make it interactive

JavaScript is a real programming language that runs right inside the browser. It makes web pages interactive. Without JavaScript, websites would be static — just text and images with no buttons, no animations, no live updates.

🧠 Think of it like this: If HTML is the structure and CSS the looks, JavaScript is the brain. It makes things happen and react to what you do — clicking a button, submitting a form, playing a video.

JavaScript started out just for browsers, but now it runs everywhere. Servers (Node.js), mobile apps (React Native), desktop apps (Electron) — all JavaScript. It's probably the most popular programming language in the world.

Variables — storing information

A variable is a named container that holds a value. Use const for values that won't change, and let for values that will:

const name = 'Alex'; // text — called a string const age = 16; // a number const loggedIn = true; // true or false — called a boolean let score = 0; // can change later score = score + 10; // now score is 10 const colours = ['red', 'green', 'blue']; // a list — called an array const user = { name: 'Alex', age: 16 }; // key-value pairs — called an object

Functions — reusable blocks of code

A function is a named block of code you can call whenever you need it:

function greet(name) { return 'Hello, ' + name + '!'; } console.log(greet('Alex')); // prints: Hello, Alex! console.log(greet('Sam')); // prints: Hello, Sam!

Making decisions — if and else

Use if to run code only when a condition is true:

const score = 75; if (score >= 55) { console.log('You passed!'); } else { console.log('Try again.'); }

What can JavaScript do?

  • React to clicks, typing, scrolling — anything you do
  • Change what's on the page without reloading it
  • Save data in the browser (like your shopping cart)
  • Send and receive data from servers (like loading new posts)
  • Check if a form is filled in correctly before sending it
  • Create animations and visual effects
  • Build games (think browser-based games!)
  • Chat in real time (like WhatsApp Web)

The DOM — how JavaScript talks to HTML

When a browser loads your HTML, it builds a tree of objects from it. This tree is called the DOM (Document Object Model). JavaScript can find, read, and change any part of that tree — that's how it makes pages interactive.

🌳 Think of it like this: The DOM is a family tree for your HTML. <html> is the root. <head> and <body> are its children. Everything inside <body> branches off from there. JavaScript can reach any branch, read it, change it, or add new ones.
// Find an element by its id attribute const title = document.getElementById('main-title'); title.textContent = 'Updated by JavaScript!'; // querySelector works like a CSS selector const btn = document.querySelector('.my-button'); // Listen for events — like a button click btn.addEventListener('click', function() { title.textContent = 'You clicked the button!'; });

Popular JavaScript frameworks

For big projects, developers use frameworks — ready-made tools that speed things up:

  • React: Made by Facebook — very widely used for building interfaces
  • Vue.js: A bit simpler than React, great for beginners
  • Angular: Made by Google — large and powerful
  • Svelte: Newer, very fast, less code to write
  • Next.js: React with extra features like server-side rendering

TypeScript — JavaScript with safety rails

TypeScript is JavaScript with one extra feature: types. You declare what kind of data each variable is allowed to hold, and TypeScript warns you before you run the code if something looks wrong.

🛡️ Think of it like this: JavaScript lets you put anything in any container. TypeScript labels each container — "numbers only", "text only". If you put the wrong thing in by accident, TypeScript catches it immediately — before the app even runs.
// JavaScript — no types, no warnings let score = 100; score = "oops, a string now"; // JavaScript is fine with this // TypeScript — types declared let score: number = 100; score = "oops, a string now"; // ❌ TypeScript warns you right away // Functions with typed parameters and return value function greet(name: string): string { return 'Hello, ' + name; }

TypeScript doesn't run directly in the browser — it gets compiled into regular JavaScript first. Most major frameworks (React, Vue, Angular) work seamlessly with TypeScript, and most professional teams use it.

💡 Should you learn TypeScript? Start with JavaScript first. Once you're comfortable with variables, functions, and the DOM, TypeScript is much easier to pick up. It's well worth learning before your first developer job.

Async JavaScript — doing things without waiting

When your browser asks a server for data, it doesn't freeze and wait. It keeps running and handles the answer when it arrives. This is called asynchronous code:

// Promises — cleaner than deeply nested callbacks fetch('/api/data') .then(response => response.json()) .then(data => console.log(data)); // Async/await (modern and cleanest) async function fetchData() { const response = await fetch('/api/data'); const data = await response.json(); return data; }

Servers & hosting

Servers — the computers that run the web

A server is a computer whose job is to wait for requests and respond to them. When your browser makes an HTTP request, it's talking to a server. The server runs the code, reads from the database, and sends back a response. Everything in this section — PHP, JSON, databases — lives and runs on a server.

🍕 Think of it like this: You're ordering pizza. The pizza place (the server) is always open, waiting for orders. You call up and place your order (the HTTP request), and they send the pizza back (the HTTP response). The pizza place doesn't call you — it waits for you to call first. And the kitchen — with all the ingredients, equipment, and recipes — is the backend you never see.

Servers are normal computers — they just run all day, every day, and have a fast internet connection. They live in data centres: large buildings with row upon row of computers, kept cool and connected to the internet at high speed.

VPS — renting your own server

Unless you work at a big tech company, you don't usually own a server. You rent one. The most common way is a VPS — a Virtual Private Server.

A VPS takes one large physical server and splits it into many smaller virtual servers. Each one acts like its own independent computer, with its own operating system, storage, and network connection. You install PHP, a database, a web server like nginx — whatever your app needs — and it runs around the clock.

  • Shared hosting: Cheapest option. Many websites share the same server. Limited control. Fine for simple sites.
  • VPS: Your own slice of a server. More control, better performance. Good for most web apps.
  • Dedicated server: An entire physical machine just for you. Expensive, but maximum power.
  • Cloud (AWS, Google Cloud, Azure): Rent as many virtual servers as you need and scale up or down instantly.
💡 Good places to start: DigitalOcean, Hetzner, and Vultr are popular VPS providers that developers recommend for their simplicity and fair pricing. A basic VPS costs around $4–6 per month.

PHP — programming on the server

PHP stands for Hypertext Preprocessor. The big difference from JavaScript is that PHP runs on the server, not in your browser. That means:

  • Visitors can never see your PHP code — they only ever get HTML back
  • You can safely connect to databases
  • You can do heavy work on the server before sending a response
  • You can show different content to different users
👨‍🍳 Think of it like this: PHP is like a chef in a restaurant. You (the browser) place your order (the request). The chef works in the kitchen (the server — fully hidden). You get your finished meal (the response). You never see what happened in the kitchen.

How PHP works, step by step

Browser sends a request
↓
Server receives it
↓
Server runs the PHP code
↓
PHP builds an HTML page
↓
Server sends that HTML to the browser
↓
Browser shows the page

PHP + HTML = dynamic pages

PHP can generate different HTML depending on who's visiting or what time it is:

<?php $currentTime = date('H:i:s'); $user = "Alex"; if ($currentTime < "12:00:00") { echo "Good morning, " . $user; } else { echo "Good afternoon, " . $user; } ?>

Security — this is really important!

Because PHP runs on the server and often talks to databases, bad security can cause serious damage. Here are the most common dangers:

⚠️ SQL Injection: A hacker types database commands into a form field to steal or destroy your data.
// WRONG — dangerous! $sql = "SELECT * FROM users WHERE email = '" . $email . "'"; // CORRECT — safe with prepared statements $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?"); $stmt->execute([$email]);
  • XSS (Cross-Site Scripting): Malicious code injected into your page — use htmlspecialchars()
  • CSRF: Fake form submissions that trick logged-in users — use tokens
  • Passwords: Never store them as plain text — always hash them with password_hash()
  • File uploads: Always check the file type and size

PHP best practices today

  • Use a framework: Laravel or Symfony — don't write everything from scratch
  • Use prepared statements: To prevent SQL injection
  • Validate all input: Never trust what users send
  • Escape output: So HTML characters don't cause problems
  • Always use HTTPS: Never send data unencrypted

JSON — the web's standard data format

JSON stands for JavaScript Object Notation. It's a simple way to write structured data as text. The cool thing about JSON is that it works with every programming language — it's not tied to JavaScript at all.

📦 Think of it like this: JSON is like a standard shipping box. No matter who packed it or who receives it, everyone knows how to open it and read what's inside. Any programming language can read JSON.

JSON is everywhere on the web. When Instagram loads your feed, when Google Maps shows you directions, when a weather app shows the forecast — that data almost certainly arrives as JSON.

Why use JSON?

  • Easy to read: Both humans and computers can understand it
  • Compact: Takes up less space than older formats like XML
  • Flexible: You can nest data inside data
  • Universal: Supported by every modern language and platform
  • Fast: Quick to parse and generate

JSON data types

JSON can hold these types of values:

{ "string": "text", "number": 42, "decimal": 3.14, "boolean": true, "null_value": null, "array": [1, 2, 3], "object": { "nested": "value" } }

A real example — what an API might send back

{ "status": "success", "data": { "id": 123, "name": "Alex", "email": "alex@example.com", "loggedIn": true, "interests": ["web", "games"], "license": { "premium": true, "expires_on": "2025-12-31" } } }

HTTP/HTTPS — how browsers and servers talk

HTTP stands for HyperText Transfer Protocol. It's the set of rules that browsers and servers use to communicate. Every time you open a website, an HTTP request happens — you just never see it.

🔒 Think of it like this: HTTP is the language your browser and the server speak to each other. HTTPS is the same language but with a digital lock — everything is scrambled so nobody can spy on it.

HTTPS is HTTP with encryption. The "S" stands for Secure. Almost all websites today use HTTPS because it protects your passwords, credit card numbers, and other private data.

What an HTTP request looks like

When you type a URL and press Enter, your browser sends something like this:

GET /page.html HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Accept: text/html,application/xhtml+xml Connection: close (optional: request body for POST)

What an HTTP response looks like

The server replies with something like this:

HTTP/1.1 200 OK Content-Type: text/html; charset=utf-8 Content-Length: 1234 Set-Cookie: session=abc123 <html>...the page content...</html>

HTTP methods — what do you want to do?

Method What it does Safe? Has body?
GETGet data from the serverYesNo
POSTSend new data to the serverNoYes
PUTReplace existing dataNoYes
PATCHUpdate part of existing dataNoYes
DELETEDelete data from the serverNoNo

HTTP status codes — what happened?

Every response has a number that tells you if things went well or not:

2xx — Success:
  • 200 OK — Everything worked fine
  • 201 Created — New item was created
  • 204 No Content — Worked fine, but nothing to send back
3xx — Redirected:
  • 301 Moved Permanently — Page has a new permanent address
  • 302 Found — Page is temporarily somewhere else
  • 304 Not Modified — Your cached version is still fine
4xx — You made a mistake:
  • 400 Bad Request — Your request doesn't make sense
  • 401 Unauthorized — You need to log in first
  • 403 Forbidden — You're logged in but not allowed
  • 404 Not Found — That page doesn't exist
5xx — The server made a mistake:
  • 500 Internal Server Error — Something crashed on the server
  • 503 Service Unavailable — Server is down or overloaded

HTTP headers — extra info on every request

Headers are like sticky notes attached to every request and response. They carry extra info:

  • Content-Type: What kind of data this is (HTML, JSON, image, etc.)
  • Content-Length: How big the data is
  • Cache-Control: Can the browser save this for later?
  • Authorization: A token proving you're logged in
  • Set-Cookie: Store a small piece of data in the browser
  • Access-Control-Allow-Origin: Which websites are allowed to use this data (CORS)

HTTP/2 and HTTP/3

Newer versions of HTTP are faster. HTTP/2 can send many requests at once. HTTP/3 is even faster, using a newer protocol called QUIC.

DNS — the internet's phone book

DNS stands for Domain Name System. Without DNS, you couldn't type "google.com" into your browser — you'd have to know the actual IP address (like 142.250.74.46). DNS translates domain names into IP addresses that computers can use.

📕 Think of it like this: DNS is like a phone book for the internet. You look up a name (google.com) and it tells you the number (the IP address). Then your browser can make the call.

DNS is one of the most important systems on the internet. Billions of DNS lookups happen every single day. If DNS goes down, huge parts of the internet stop working.

How DNS works, step by step

1. Browser checks its own cache first
↓
2. Asks the Recursive Resolver (usually your ISP)
↓
3. Resolver asks a Root Nameserver
↓
4. Root says: "Ask the TLD nameserver for .com"
↓
5. TLD says: "Ask the authoritative nameserver for that domain"
↓
6. Authoritative server gives back the IP address
↓
7. Browser gets the IP and connects!

DNS record types

  • A Record: Links a domain to an IPv4 address (like 192.0.2.1)
  • AAAA Record: Links a domain to an IPv6 address (the newer, longer format)
  • CNAME: An alias — "this domain is the same as that one"
  • MX Record: Which server handles email for this domain
  • TXT Record: Text info — used for verification, spam protection, etc.
  • NS Record: Which nameserver is responsible for this domain

DNS caching — saving lookups for later

DNS lookups take time, so results are saved (cached) at several points:

  • Browser cache: Your browser remembers recent lookups for a few minutes
  • OS cache: Your computer also saves lookups
  • ISP cache: Your internet provider caches them too
  • TTL (Time To Live): How long a result can be cached — set by the domain owner

DNS security

  • DNSSEC: Adds digital signatures to prevent fake DNS responses
  • DoH (DNS over HTTPS): Encrypts DNS lookups so no one can spy on which sites you visit
  • DoT (DNS over TLS): Same idea, different method

APIs — how apps talk to each other

API stands for Application Programming Interface. An API is a set of rules for how two programs can communicate. One program says "give me this", and the API defines exactly what comes back.

🍽️ Think of it like this: An API is like a menu in a restaurant. You pick what you want from the menu (the API). The kitchen (the server) makes it and sends it out. You don't need to know how it was cooked — just what to order.

APIs are everywhere. When your weather app shows the forecast, it's calling a weather API. When you log in with Google, that's OAuth — a type of API. When a map loads in an app, that's the Google Maps API.

Why are APIs useful?

  • Abstraction: You don't need to understand the internals — just the interface
  • Reusability: Many apps can use the same API
  • Security: You only expose what you want to share
  • Flexibility: The inside can change without breaking the outside
  • Standard: Everyone speaks the same language

REST APIs — the most common style

REST (Representational State Transfer) is the most popular way to design web APIs. A good REST API follows these ideas:

  • Resources: Everything is a "thing" you can address, like /users or /posts
  • HTTP Methods: GET to read, POST to create, PUT/PATCH to update, DELETE to remove
  • Stateless: Each request contains all the info needed — no memory between requests
  • JSON: Data is usually sent and received as JSON
  • Status codes: Use the right HTTP status code for each response

A real example — Twitter API

GET /2/tweets/:id → Get a specific tweet POST /2/tweets → Post a new tweet DELETE /2/tweets/:id → Delete a tweet GET /2/users/:id/following → Get who this user follows

Authentication — proving who you are

Most APIs require you to prove your identity before they'll respond:

  • API Keys: A secret code you include in every request
  • OAuth 2.0: The standard used by Google, Facebook, GitHub, etc.
  • JWT (JSON Web Tokens): A small token that proves who you are
  • HTTP Basic Auth: Username and password (only safe over HTTPS)

GraphQL — a newer alternative

GraphQL lets you ask for exactly the fields you need — no more, no less. This avoids downloading data you don't use:

query { user(id: 1) { name email posts { title date } } }

Rate limiting — don't overload the server

Most APIs limit how many requests you can make. If you go over the limit, you get a 429 error. A typical limit might be 1000 requests per hour.

Databases — storing data permanently

A database is an organized way to store lots of data so you can find it again quickly. Your user account on any website, your saved posts, your order history — all of that lives in a database.

📚 Think of it like this: A database is like a super-fast, super-smart spreadsheet. You can store millions of rows and search through them in milliseconds. Try that in Excel!

SQL vs NoSQL — two different approaches

There are two main types of databases, and they work very differently:

SQL (Relational databases) — structured and strict:
  • Structure: Tables with rows and columns, like a spreadsheet
  • Schema: You must define the columns before you can add data
  • ACID: Transactions are reliable and safe
  • Examples: MySQL, PostgreSQL, SQLite, SQL Server
  • Good for: Bank accounts, orders, anything where accuracy matters
NoSQL (Document databases) — flexible and fast:
  • Structure: Documents (think JSON objects)
  • Schema: Flexible — each document can have different fields
  • Scalable: Easy to spread across many servers
  • Examples: MongoDB, Firebase, CouchDB
  • Good for: User profiles, content, logs, anything that changes a lot

SQL in action

-- Get all users SELECT * FROM users; -- Filter and sort SELECT name, email FROM users WHERE city = 'Amsterdam' ORDER BY name; -- Count users per city SELECT city, COUNT(*) FROM users GROUP BY city; -- Join two tables together SELECT users.name, posts.title FROM users JOIN posts ON users.id = posts.user_id;

Indexes — making searches faster

An index is like the index at the back of a book. Instead of reading every page to find something, you jump straight to it. Indexes make database searches much faster, but they take up extra space.

  • Primary Key: A unique ID for every row
  • Foreign Key: A link to a row in another table
  • Trade-off: Faster to read, a little slower to write

Database security

  • Backups: Always back up your data!
  • Access control: Not everyone should be able to read or change everything
  • Encryption: Encrypt sensitive data like passwords
  • Parameterized queries: Protect against SQL injection attacks

Frontend vs backend — the two halves of a web app

Every professional web application has two main parts. They run on different computers and talk to each other over the internet.

🏪 Think of it like this: Frontend is the shop floor — what customers see and interact with. Backend is the stockroom — where all the real work happens, completely out of sight.

Frontend — what runs in your browser

Technologies: HTML, CSS, JavaScript, React, Vue, Angular
Runs on: Your device (phone, laptop, etc.)
What it does:
  • Shows the interface — buttons, menus, text, images
  • Responds to clicks and typing instantly, no waiting
  • Checks form input before sending it to the server
  • Sends requests to the backend
  • Displays the data it gets back (usually JSON)
  • Can work offline with service workers
Advantages:
  • Fast — reacts instantly without asking the server
  • Can work offline
  • Less load on the server
Limitations:
  • Anyone can read your JavaScript code
  • Can't talk directly to a database
  • Limited by the user's device

Backend — what runs on the server

Technologies: PHP, Python, Node.js, Java, Go, Rust
Runs on: A server in a data center
What it does:
  • Receives requests from the frontend
  • Runs the business logic (checks, calculations, rules)
  • Reads and writes to the database
  • Sends JSON back to the frontend
  • Keeps code and data private and secure
Advantages:
  • Code is completely hidden from users
  • Can scale — add more servers if needed
  • Direct access to the database
  • Can do heavy tasks (send emails, process images)
Limitations:
  • Every action involves a network request (slower)
  • Costs money to host
  • If it goes down, everything stops

The full journey — a login, step by step

Frontend (Browser)
1. User fills in login form
2. JavaScript checks the email and password look valid
3. JavaScript sends HTTPS POST to the server

↓ (travels over the internet) ↓

Backend (Server)
4. PHP receives the request
5. PHP looks up the email in the database
6. PHP checks the password hash (never stores plain text!)
7. PHP creates a session token
8. PHP sends JSON back: { "success": true, "token": "..." }

↓ (travels over the internet) ↓

Frontend (Browser)
9. JavaScript reads the JSON
10. JavaScript saves the token
11. JavaScript redirects to the dashboard
12. User sees their dashboard!

API-first — the modern approach

Most modern apps are built "API-First": the backend builds a clean, documented API. The frontend, mobile app, and any third-party apps all use that same API. This means the website and the mobile app don't duplicate logic — they share it.

Responsive design — one website, all devices

Responsive design means your website looks and works great on every screen — a small phone (320px wide), a tablet, a laptop, a big monitor. More than 60% of web traffic now comes from phones, so this really matters.

🪑 Think of it like this: Responsive design is like adjustable car seats. The same seat fits a small child and a tall adult. A responsive website fits any screen size just as comfortably.

The three pillars of responsive design

1. Flexible Layouts
  • Use percentages instead of fixed pixel widths
  • Use CSS Grid or Flexbox to arrange things
  • Elements grow and shrink with the screen
2. Flexible Media
  • Images resize automatically (max-width: 100%)
  • Videos also adapt to screen size
  • Use srcset to send high-res images only to high-res screens
3. Media Queries
  • Write different CSS for different screen sizes
  • Common breakpoints: 480px (phone), 768px (tablet), 1024px (desktop)
  • Can also target landscape vs portrait orientation

Mobile-first — start small, build up

The modern approach is to design for mobile first, then add styles for bigger screens:

/* MOBILE — the base styles */ body { font-size: 14px; padding: 10px; } .container { display: block; /* stack vertically on small screens */ } /* TABLET */ @media (min-width: 768px) { body { font-size: 16px; } .container { display: grid; grid-template-columns: 1fr 1fr; } } /* DESKTOP */ @media (min-width: 1024px) { body { font-size: 18px; max-width: 1200px; margin: 0 auto; } .sidebar { width: 250px; } }

Responsive best practices

  • Viewport meta tag: Always include <meta name="viewport" content="width=device-width">
  • Touch-friendly: Buttons should be at least 44x44px — fingers aren't as precise as mouse cursors
  • Readable text: Use at least 16px on mobile
  • Compress images: Mobile connections are slower — don't send huge files
  • Test on real devices: Emulators are good but real phones are better
  • No horizontal scrolling: If your page scrolls sideways, something's wrong

Tools for testing responsive design

  • Browser DevTools: Press F12, then click the phone icon to simulate different screens
  • Google Mobile-Friendly Test: Check if Google sees your page as mobile-friendly
  • Lighthouse: Built into Chrome — tests performance, accessibility and SEO
  • BrowserStack: Test on real devices in the cloud

Everything together — the full picture

A full example: buying something in an online shop

You type a URL into your browser
↓
DNS looks it up: "That domain is at 192.0.2.1"
↓
Browser sends: HTTPS GET /index.html
↓
Server sends back: HTML + CSS + JavaScript
↓
Browser builds and shows the page
↓
You scroll — JavaScript loads products
↓
JavaScript sends: GET /api/products (HTTPS)
↓
PHP receives the request on the server
↓
PHP fetches products from the Database
↓
PHP sends JSON: [{id: 1, name: "Product", price: 9.99}, ...]
↓
JavaScript shows the products on screen
↓
You click "Buy"
↓
JavaScript sends: POST /api/order {product: 1, quantity: 2}
↓
PHP checks the order and saves it to the Database
↓
PHP sends: {status: "success", orderNumber: 12345}
↓
JavaScript shows a success message!

A suggested learning path

  1. Month 1: HTML + CSS basics — build a simple static website
  2. Month 2: JavaScript basics — add buttons, forms, and interactions
  3. Month 3: Responsive Design — make it work on a phone
  4. Month 4: HTTP and DNS — understand what's happening under the hood
  5. Month 5: PHP basics — learn to write server-side code
  6. Month 6: Databases — learn to store and query data with MySQL
  7. Month 7: Build APIs — connect your frontend to your backend
  8. Month 8: Build your own project — put it all together
  9. Month 9+: Learn React or Vue — modern frontend development
